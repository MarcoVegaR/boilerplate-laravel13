# ADR-007: Logging and Correlation Policy

**Status**: Vigente  
**Date**: 2026-03-21  
**Authors**: Caracoders Engineering

**Originating PRD**: PRD-03, reconciled by PRD-07  
**Implementation references**: `app/Http/Middleware/HandleCorrelation.php`, `bootstrap/app.php`, `config/logging.php`, `app/Services/SecurityAuditService.php`, `routes/console.php`

---

## Context

The boilerplate requires end-to-end traceability of requests and operations across HTTP, queue workers, and background jobs. Without a consistent correlation mechanism, debugging production incidents is limited to timestamp-based log searches with no way to group related log entries.

---

## Decision

### Correlation ID mechanism

Every HTTP request is assigned a `correlation_id` (UUID v4) by `App\Http\Middleware\HandleCorrelation`.

- If the request carries a valid UUID in `X-Correlation-ID`, that value is reused.
- If the header is absent or invalid, a new UUID is generated.
- The correlation ID is stored using `Context::add('correlation_id', $value)` and returned in the `X-Correlation-ID` response header.

### `Context` facade (NOT `Log::shareContext()`)

The `Context` facade (`Illuminate\Support\Facades\Context`) is used instead of `Log::shareContext()`. Key reasons:

- `Context` data propagates **automatically to all queued jobs** dispatched within the same request. This means a job dispatched from a correlated request inherits `correlation_id` without any extra code.
- `Log::shareContext()` does NOT propagate to queued jobs — it is request-scoped only.
- `Context::add()` is the canonical Laravel 11+ way to set shared context for the full lifecycle.

### Channel strategy

| Channel    | Driver | Path                              | Purpose                                    |
| ---------- | ------ | --------------------------------- | ------------------------------------------ |
| `stack`    | stack  | (channels: `daily`)               | Default application log                    |
| `daily`    | daily  | `storage/logs/laravel-{date}.log` | General app logs, rotated daily            |
| `security` | daily  | `storage/logs/security.log`       | Security events: login, logout, 2FA, roles |

The `security` channel is isolated from the main `stack` by design. Security events must not be mixed with application debug logs, and operational teams may ship `security.log` to a SIEM independently.

### Correlation header convention

- **Request header**: `X-Correlation-ID` (incoming from caller or generated by middleware).
- **Response header**: `X-Correlation-ID` (always echoed back — enables client-side tracing).
- **Log context key**: `correlation_id` (set via `Context::add()`).
- All log entries for a given request share the same `correlation_id` via the Context facade.

### `LOG_FORMAT=json` note

Setting `LOG_FORMAT=json` in `.env` activates JSON log formatting for the `daily` channel. This is documented in `.env.example` as a comment (`# LOG_FORMAT=json`). JSON format is required for log aggregation tools (e.g., Datadog, Elastic, Loki). This requires a corresponding handler tap or formatter in `config/logging.php` — see the commented example in that file.

### Additional context shared per request

The middleware also adds:

- `user_id` — authenticated user's ID, or `null` for guests.
- `url` — HTTP method + path (e.g., `GET dashboard`).

### Middleware registration

`HandleCorrelation` is the **first** middleware appended to both `web` and `api` groups in `bootstrap/app.php`. It must run before any code that might write log entries.

## Logging prohibitions

The following patterns are **prohibited** in this codebase:

| Prohibited                                     | Reason                                          | Correct alternative                                                  |
| ---------------------------------------------- | ----------------------------------------------- | -------------------------------------------------------------------- |
| `env('LOG_*')` outside config files            | Bypasses config layer                           | Read from `config('logging.*')`                                      |
| `Log::withContext()` for correlation           | Request-scoped only — doesn't propagate to jobs | Use `Context::add()` in middleware                                   |
| `Log::shareContext()` for correlation          | Same issue as above                             | Use `Context::add()`                                                 |
| `Log::emergency()` for business errors         | Reserved for system-level alerts                | Use `BoilerplateException` or `Log::warning()`                       |
| Plain `Log::info('login')` for security events | Not isolated, no structure                      | Use `Log::channel('security')->info(...)` via `SecurityAuditService` |

---

## Consequences

- ✅ Every log entry in production can be correlated back to a single HTTP request or job dispatch.
- ✅ Correlation propagates to queue workers automatically via `Context`.
- ✅ Security events are isolated in a dedicated log channel.
- ✅ The `X-Correlation-ID` response header enables client-side and API gateway tracing.
- ⚠️ `LOG_FORMAT=json` requires configuration of the formatter — the `.env.example` comment documents this as an optional activation.
- ⚠️ Jobs dispatched outside of an HTTP context (scheduler, artisan commands) will not have a pre-existing `correlation_id` — they should generate one in their `handle()` method if needed.
